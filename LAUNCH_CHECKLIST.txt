# Celestia 6-Week Launch Checklist

## Week 1: Foundation & Setup

### Development Environment
- [ ] All team members have local development environment running
- [ ] Docker Compose successfully starts all services (Postgres, Redis, Backend)
- [ ] Mobile app runs on iOS simulator and Android emulator
- [ ] Database migrations execute successfully
- [ ] Seed data loads correctly (360 horoscopes)

### Code Review & Quality
- [ ] All code passes linting (no errors, warnings acceptable with justification)
- [ ] TypeScript strict mode enabled and all type errors resolved
- [ ] Unit test coverage > 70% for backend
- [ ] Unit test coverage > 60% for mobile app
- [ ] Integration tests passing for all API endpoints

### Documentation
- [ ] README.md complete with setup instructions
- [ ] API documentation generated (Swagger/OpenAPI)
- [ ] privacy_safety.md reviewed by legal/compliance consultant
- [ ] Code comments added for complex algorithms (especially distress detection)

### Security Baseline
- [ ] All secrets moved to environment variables (no hardcoded credentials)
- [ ] .env.example files created for all projects
- [ ] npm audit shows no high/critical vulnerabilities
- [ ] SQL injection protection verified (parameterized queries only)
- [ ] XSS protection enabled in Express (helmet middleware)

## Rollback Plan

If critical issues arise post-launch:

1. **Severity Assessment** (5 minutes):
   - Critical: App crashes, data loss, security breach → Immediate rollback
   - High: Feature broken, poor performance → Fix forward or rollback within 1 hour
   - Medium: UI issues, minor bugs → Fix in next release

2. **Rollback Procedure** (15 minutes):
   ```bash
   # Backend rollback
   aws ecs update-service \
     --cluster celestia-production \
     --service celestia-backend-production \
     --task-definition celestia-backend:[PREVIOUS_VERSION]
   
   # Database rollback (if migration issue)
   npm run migrate:rollback
   ```

3. **Communication**:
   - Update status page immediately
   - Notify users via in-app banner if service degraded
   - Post to social media if widespread issue
   - Send post-mortem within 48 hours

## Risk Register

### High-Risk Items

| Risk | Impact | Mitigation | Owner |
|------|--------|-----------|-------|
| Distress detection false positives annoy users | High | A/B test threshold, allow users to dismiss | Product |
| App store rejection for astrology/pseudoscience | High | Emphasize entertainment disclaimer, show safety features | Legal/Product |
| Viral growth overwhelms infrastructure | High | Auto-scaling configured, load testing done | DevOps |
| Data breach exposes PII | Critical | Encryption at rest, regular security audits, bug bounty | Security |
| Mental health crisis not handled properly | Critical | Mental health professional on retainer, clear escalation paths | Operations |

### Medium-Risk Items

| Risk | Impact | Mitigation | Owner |
|------|--------|-----------|-------|
| Horoscope content becomes repetitive | Medium | Content calendar, multiple writers, user feedback | Content |
| Premium features insufficient value | Medium | User research, iterate on premium offering | Product |
| Localization challenges for non-English markets | Medium | Start with English-only, expand based on demand | Product |
| Competition from established horoscope apps | Medium | Differentiate on safety/privacy, superior UX | Marketing |
